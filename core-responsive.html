Core Foundation

Responsive Strategy

Context-driven responsive design: why surfaces adapt differently and how components transform.

Philosophy

Responsive design at Saltwyk is driven by user context, not screen size. A 768px viewport could be a tablet held vertically, a desktop browser resized, or a laptop in split-screen mode. The breakpoint tells us nothing about the user's intent or environment. Context does.

We design for where and how people use each surface. A shopper completing checkout is likely on a phone, thumb hovering, standing in a store or sitting on a couch. A merchant reviewing transactions is at a desk with a mouse and keyboard, perhaps with multiple windows open. The device follows behavior, not the reverse.

This means different surfaces have different responsive strategies. Shopper Passport is mobile-first because that's where shoppers live. Merchant Dashboard is desktop-first because that's where operators work. Neither approach is universally correct—each is correct for its context.

Surface Contexts

Each surface has a primary user context that determines its responsive strategy.

Shopper Passport

my.saltwyk.com

Mobile-First
Primary context Mobile (phone in hand at checkout)
Secondary context Desktop (merchant team testing, employees as shoppers)
Design approach Mobile-first
Why mobile-first Shoppers encounter Saltwyk at point of purchase, usually on a merchant's mobile site or in-store. The checkout moment is handheld.
Desktop consideration Merchant team members working at desks are also shoppers. They need to test the flow and may use their own points. Desktop must work, but mobile is the design origin.

Checkout Widget

Embedded in merchant checkout

Container-Responsive
Primary context Embedded in merchant checkout (any device)
Design approach Container-responsive
Why container-responsive Widget lives inside merchant sites we don't control. Must adapt to available width, not viewport.
Constraints min-width: 320px, max-width: 480px. Designed for constraint.

Merchant Dashboard

dashboard.saltwyk.com

Desktop-First
Primary context Desktop (operational work)
Secondary context Tablet (possible, not prioritized)
Design approach Desktop-first
Why desktop-first Merchants manage their business at workstations. Data density, multi-column layouts, and keyboard shortcuts require screen real estate.
Mobile consideration Basic functionality should work on tablet for quick checks, but this is not the primary design target.

Marketing Website

saltwyk.com

Responsive (Both)
Primary context Desktop and mobile equally
Design approach Responsive (both breakpoints designed intentionally)
Why both Prospects arrive from anywhere—LinkedIn on desktop, social ads on mobile. Neither is secondary.

Documentation

docs.saltwyk.com

Desktop-First
Primary context Desktop (coding at workstation)
Secondary context Mobile (quick reference)
Design approach Desktop-first with mobile fallback
Why desktop-first Developers implement while reading docs. Code editor + docs side-by-side is the primary use case.
Mobile consideration Must be readable for quick lookups, but not optimized for extended reading.

Touch vs. Pointer

Touch Targets (Mobile)

  • 44

    Minimum: 44×44px

    Apple HIG minimum tap target

  • 48

    Recommended: 48×48px

    Comfortable thumb target

  • Spacing: 8px minimum

    Between adjacent targets

Pointer Targets (Desktop)

  • 32

    Minimum: 32×32px

    Precise pointer allows smaller targets

  • Can be denser than mobile

    Data density is acceptable

  • Hover states provide affordance

    Visual feedback on cursor proximity

Adaptation Guidance

Element Touch Adaptation
Buttons Same visual size, but increase padding/hit area on touch
Links Inline on desktop, larger tap targets on mobile
Dropdowns Click-to-open on desktop, tap-to-open on mobile (no hover-to-open)

Testing Checklist

Responsive QA requirements by surface.

Mobile-First

Shopper Passport

  • Works on 320px width (small phones)
  • Primary flow completable with thumb only
  • Touch targets meet 44px minimum
  • Desktop view is functional (not broken)
Container

Checkout Widget

  • Renders correctly at 320px container
  • Renders correctly at 480px container
  • No horizontal overflow
  • Form inputs are usable on mobile
Desktop-First

Merchant Dashboard

  • Full functionality at 1024px+
  • Graceful degradation at 768px (tablet)
  • Critical actions accessible on mobile
Both

Marketing Website

  • Hero sections work at all breakpoints
  • CTAs are prominent on mobile
  • Feature sections stack appropriately
  • Typography scales are readable
