Core Foundation

Dual-Context Responsive Strategy

Why surfaces adapt differently: designing for primary device contexts while intentionally supporting secondary.

Philosophy

Saltwyk serves two distinct contexts ‚Äî shoppers managing their rewards and merchants running their business ‚Äî often for the same person. Rather than a single responsive approach, we design intentionally for each context's primary device while ensuring secondary devices work well, not as an afterthought.

Every surface has a primary and secondary device context. We design for the primary first, then intentionally adapt for secondary ‚Äî never just letting layouts "flow" responsively.

Key principle: A breakpoint tells us nothing about user intent. Context does. A merchant reviewing transactions is at a desk with a keyboard. A shopper completing checkout is standing in a store with a phone. The device follows behavior, not the reverse.

Adaptation Principles

Primary Context

  • ‚Ä¢ Full feature set
  • ‚Ä¢ Optimized UX for the device
  • ‚Ä¢ This is where we invest the most design effort

Secondary Context

  • ‚Ä¢ Core features (not necessarily all features)
  • ‚Ä¢ Native-feeling to the device (not crammed or stretched)
  • ‚Ä¢ Intentionally designed, not just "responsive"

Never

  • ‚Ä¢ Let mobile be "squeezed desktop"
  • ‚Ä¢ Let desktop be "blown-up mobile"
  • ‚Ä¢ Assume responsive CSS alone creates good adaptation

Component Behavior Across Contexts

Core components are shared across all surfaces. Their responsive behavior is consistent, but their density and spacing may vary by context.

Shared Behaviors

  • ‚úì

    Buttons

    Same sizes, touch targets (min 44px) on all devices

  • ‚úì

    Inputs

    Same heights, full-width on mobile, constrained on desktop

  • ‚úì

    Cards

    Stack vertically on mobile, grid on desktop

  • ‚úì

    Navigation

    Persistent on desktop, collapsed (hamburger/bottom nav) on mobile

Context-Specific Density

Shopper Surfaces

Comfortable spacing (more generous padding, larger touch targets, breathing room)

Merchant Surfaces

Compact spacing option available (information density, more data per screen)

Both use the same spacing scale, just different defaults. Merchants can toggle between comfortable and compact modes.

Implementation Notes

When building a new surface: First identify its primary device context. Design for that context first, then intentionally adapt for secondary. Review against the testing matrix before shipping.
