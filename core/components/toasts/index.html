Component Specification

Toast Notifications

Implementation specification for Saltwyk toast components. Every state, variant, and pixel value defined.

Section 1

Toast Philosophy

Toasts provide non-blocking feedback for completed actions. They appear briefly and dismiss automatically (or manually), informing users without interrupting their workflow. Unlike modals, toasts don't demand immediate attention—they inform alongside ongoing tasks.

Toast is for...

  • • Action confirmations ("Settings saved")
  • • Non-critical errors ("Retry available")
  • • Background process updates
  • • Information that can be missed

Toast is NOT for...

  • • Decisions requiring user action
  • • Critical errors that block work
  • • Persistent warnings
  • • Form validation feedback

Bright Toast (Fire Opal Emphasis)

For critical messages that must stand out on dark backgrounds. Use sparingly—for critical success or error only.

Type Background Icon Title Message

Design Decision: Limited Bright Variants

Only Success and Error have bright variants. Warning and Info don't warrant the same level of visual urgency. Bright toasts should be reserved for critical confirmations (large payments) or critical failures (transaction declined).

Section 11

Animation

Enter Animation

/* Slide in from right */ @keyframes toast-enter-right { from { opacity: 0; transform: translateX(100%); } to { opacity: 1; transform: translateX(0); } }

Duration: 200ms, ease-out

Exit Animation

/* Slide out to right */ @keyframes toast-exit-right { from { opacity: 1; transform: translateX(0); } to { opacity: 0; transform: translateX(100%); } }

Duration: 150ms, ease-in

Reduced Motion

/* Respect prefers-reduced-motion */ @media (prefers-reduced-motion: reduce) { .toast-enter { animation: toast-fade-in 150ms ease-out; } .toast-exit { animation: toast-fade-out 100ms ease-in forwards; } }

Fade only, shorter duration for users who prefer reduced motion.

Section 12

Toast vs. Alert Banner vs. Modal

Choose the right pattern for your feedback needs.

Pattern Blocking Duration Use Case
Toast No Temporary (3-8s) Feedback for completed actions
Alert Banner No (but persistent) Until dismissed Page-level warnings, system status
Modal Yes Until user acts Confirmations, decisions, forms

Use Toast when...

  • • Action completed successfully
  • • Non-critical error (can retry)
  • • Informational update
  • • Can be missed without consequence

Use Alert Banner when...

  • • Persistent warning needed
  • • System status (offline mode)
  • • Page-level error affecting content
  • • Must not be missed

Use Modal when...

  • • User must make a decision
  • • Confirmation before destructive action
  • • Complex info needing attention
  • • Form input required

Section 14

Accessibility

ARIA Requirements

role="status" or role="alert"

Use alert for errors, status for others

aria-live="polite" or "assertive"

Use assertive for errors only

aria-atomic="true"

Announce entire toast content

Focus Management

  • • Toasts should NOT steal focus from current task
  • • Dismiss button must be focusable if toast persists
  • • Action buttons/links must be keyboard accessible
  • • aria-label="Dismiss notification" on close button

Screen Reader Behavior

  • • Toast content announced when it appears
  • • Auto-dismiss should not interrupt current task
  • • Important actions should remain available

Motion Sensitivity

  • • Respect prefers-reduced-motion
  • • Reduce or eliminate slide animations
  • • Keep fade, reduce duration
  • • Progress bar can still animate (subtle)

  • Error: magenta-500 left border
  • Info: indigo-500 left border

Dark Mode

  • Background: intaglio-800
  • Border: intaglio-700 + colored left border
  • Title: warm-100
  • Message: warm-400

Animation

  • Enter: 200ms slide + fade
  • Exit: 150ms slide + fade
  • Reduced motion: fade only, shorter

Auto-dismiss

  • Default: 5 seconds
  • Progress bar: 3px, semantic color @ 30%
  • Pause on hover (optional)

Stacking & Accessibility

  • Position: top-right default
  • Gap: 12px between toasts
  • role="status" or role="alert"
  • aria-live="polite" or "assertive"
